This comprehensive tutorial demonstrates how to create and configure cross-chain tokens using Chainlink's Cross-Chain Interoperability Protocol (CCIP) between Solana Devnet and Ethereum Sepolia. You will implement the **direct mint authority transfer** approach within **Path A** from the [CCIP Cross-Chain Token Integration Guide](/ccip/concepts/cross-chain-token/svm/integration-guide).

## What You Will Build

This tutorial implements the **direct mint authority transfer** variant of **Path A** from the [CCIP Cross-Chain Token Integration Guide](/ccip/concepts/cross-chain-token/svm/integration-guide). This approach is designed for development and testing environments where you transfer complete mint authority to the Pool Signer PDA for simplified setup.

| Component                | Implementation                      | Authority Model           |
| ------------------------ | ----------------------------------- | ------------------------- |
| **Ethereum Sepolia**     | ERC20 token with CCIP BurnMint pool | EOA-controlled            |
| **Solana Devnet**        | SPL token with CCIP BurnMint pool   | Direct PDA mint authority |
| **Cross-Chain Transfer** | Bidirectional token transfers       | Autonomous operations     |

**Key Implementation Points:**

- **Mint Authority Control**: You control the token mint authority initially
- **Self-Service Registration**: Use `owner_propose_administrator` directly
- **BurnMint Pools**: Burn tokens on source, mint on destination
- **Direct Transfer**: Complete mint authority transferred to Pool Signer PDA

<Aside type="caution" title="Development Environment Approach">
  **Important**: This tutorial transfers mint authority directly to the Pool Signer PDA for simplicity. This approach is
  designed for **development and testing environments only**. **For production deployments**, consider using: - [SPL
  Token Multisig Tutorial](/ccip/tutorials/svm/cross-chain-tokens/spl-token-multisig-tutorial) - Educational multisig
  concepts - [Production Multisig Tutorial](/ccip/tutorials/svm/cross-chain-tokens/production-multisig-tutorial) -
  Enterprise-grade dual-layer governance See [Mint Authority
  Management](/ccip/concepts/cross-chain-token/svm/token-pools#mint-authority-management) for detailed security
  considerations.
</Aside>

## Technical Architecture

### Cross-Chain Token Mechanism

The BurnMint mechanism ensures total supply consistency:

1. **Source Chain**: Burns tokens from sender's account
1. **CCIP Protocol**: Transmits message cross-chain
1. **Destination Chain**: Mints equivalent tokens to receiver

**This is the defining step** of the direct mint authority approach. You're transferring complete mint authority to
the Pool Signer PDA. This enables autonomous cross-chain operations but removes your direct control over minting.
**Before proceeding**, ensure you've minted any tokens needed for testing, as you won't be able to mint directly after
this step.

</Aside>

- You cannot directly mint tokens anymore
- No governance mechanism to recover mint authority

**For production**, consider [multisig approaches](/ccip/tutorials/svm/cross-chain-tokens/production-multisig-tutorial) that maintain governance control.

</Fragment>
</Tabs>

## Phase 3: Cross-Chain Configuration

Configure bidirectional connectivity between your token pools on both chains.

### Understanding PDA Formats

<Aside type="note" title="PDA Format Conversion">

**Important Concept**: Solana uses base58 encoding for addresses, while Ethereum uses hexadecimal. When configuring cross-chain settings:

- **Solana operations**: Use base58 PDA format (e.g., `CB9NEes1KzH3WmsnXA1bH3Qyu3gjaKJMfyGNVgtVZw8e`)
- **EVM configuration**: Convert to 32-byte hex format (e.g., `0xabc123...`)

The system handles most conversions automatically, but understanding this distinction helps with troubleshooting.

</Aside>
