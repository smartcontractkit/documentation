This page consolidates recommended practices and known risks / remediation for securing Cross-Chain Token (CCT) deployments on CCIP. Items are grouped by topic: Critical/Warning/Info labels are added to relevant sections to indicate relative priority—it is recommended that token developers review and apply them based on their own independent risk assessment. It is also recommended to utilize active monitoring and logging to ensure that the security of your CCT is maintained. For additional assistance with risk assessment or to discuss your specific use case, [contact Chainlink Labs](https://chain.link/ccip-contact).

For chain-family-specific integration details, see the [Cross-Chain Token Standard](/ccip/concepts/cross-chain-token) section. For rate limit operations, see [Rate Limit Management](/ccip/concepts/rate-limit-management/overview).

<h2 id="secure-signing-and-opsec">Secure signing and opsec</h2>

Addresses controlled by a single private key or EOAs are generally not recommended to control roles in CCT-related contracts that could lead to a loss of funds or other critical issues such as: contract ownership, burn/mint rights, ccip administrator. It is **highly recommended** to use a secure solution such as a multi-signature smart account or timelock to avoid a single point of failure.

Different roles may warrant **varied signing setups**. High-privilege changes (ownership transfer, pool upgrades, registry updates) typically benefit from a broad multisig and timelock. **Scoped** actions for risk management and incident response may use a separate, more responsive configuration. For example, a Token Developer may choose to assign the [`rateLimitAdmin`](/ccip/concepts/rate-limit-management/prerequisites-and-permissions) role on the token pool to an address that can pause or throttle a lane via rate limit configuration without holding full pool owner rights. See [Emergency Actions](/ccip/concepts/rate-limit-management/emergency-actions) and [Rate limit admin role](#rate-limit-admin-role) below.

Review every address in a multisig regularly to confirm signers remain trusted and that departed or compromised keys are removed promptly. It is also recommended to balance security with operational needs: choose a threshold, geographic distribution, and approval workflow that supports both routine governance and timely incident response (for example: pausing a lane through the rate limit admin).

**Recommendations:**

- Use multi-signature smart accounts (for example, [Safe Smart Accounts](https://github.com/safe-global/safe-smart-account)) with a threshold of **at least 3-of-n** signers commensurate with risk.
- Add a **timelock** with propose, review, and cancel steps so critical actions cannot execute immediately without oversight.
- Remove deprecated or compromised signers promptly and audit signer access quarterly.
- Create runbooks for routine operations and incident response workflows that support both routine governance, configuration changes, and timely incident response.

**Related docs:** [CCT Administration (EVM)](/ccip/concepts/best-practices/evm#best-practices-for-cross-chain-token-cct-administration), [Multi-Signature Contracts](/ccip/concepts/best-practices/evm#multi-signature-contracts), [Token Manager](/ccip/tools-resources/token-manager), [EVM Upgradability (MCMS)](/ccip/concepts/architecture/onchain/evm/upgradability)

## Pool and lane configuration

<h3 id="incorrect-pool-mappings">Incorrect pool mappings (Critical)</h3>

Cross-chain lanes require symmetric, correct configuration on **both** sides of every lane.

**Validation checklist (EVM):**

- Source `remotePoolAddresses` must match the destination token pool address.
- Source `remoteTokenAddress` must match the token returned by `getToken()` on the destination pool.
- The destination chain must have a token pool registered via `TokenAdminRegistry.setPool`.
- Both pools must call [`applyChainUpdates`](/ccip/concepts/cross-chain-token/evm/registration-administration#applychainupdates) with matching chain selectors, addresses, and rate limits.

See [Pool Configuration Checklist](/ccip/concepts/cross-chain-token/evm/registration-administration#pool-configuration-checklist).

<Aside type="note" title="SVM">
  On Solana, configure remote pool addresses and token mints via pool program instructions. See [SVM Registration &
  Administration: Remote Chain
  Configuration](/ccip/concepts/cross-chain-token/svm/registration-administration#remote-chain-configuration).
</Aside>

<h3 id="cross-silo-isolation">Cross-silo isolation (Critical)</h3>

If your token uses **siloed** liquidity lanes (isolated hub-and-spoke topology), do not connect siloed lanes to networks outside their designated hub chain.

**Risk:** Cross-silo connections cause incorrect accounting, stuck transactions, or loss of funds.

**Remediation:** Remove unintended remote chains from the token pool using `applyChainUpdates` (EVM) or the equivalent remote config update on your chain family.

<h3 id="decimal-symmetry">Decimal symmetry (Critical)</h3>

Token decimals should match across chains when possible. When they differ, configure Token Pool Rate Limits (TPRLs) using the correct decimal model for your TokenPool version.

**Related docs:** [Token Decimal Handling (EVM)](/ccip/concepts/cross-chain-token/evm/token-pools#token-decimal-handling), [Token Units and Decimals](/ccip/concepts/rate-limit-management/token-units-and-decimals), [SVM Decimal Compatibility](/ccip/concepts/cross-chain-token/svm/token-pools#decimal-compatibility-considerations)

<h3 id="supply-imbalance">Supply imbalance (Critical)</h3>

For **Lock and Mint** architectures, tokens locked in the LockRelease pool on the issuing chain must be **greater than or equal to** the total supply minted on remote BurnMint chains.

**Risk:** A token shortfall can cause transfer failures or require manual infusion to make users whole.

**Remediation:** Regularly compare `balanceOf(pool)` on the home chain against the sum of `totalSupply` on all remote mint chains. See [Liquidity Management](/ccip/concepts/best-practices/evm#best-practices-for-liquidity-management).

## Token pool rate limits

<h3 id="unset-token-pool-rate-limits">Unset token pool rate limits (Critical)</h3>

Inbound or outbound TPRLs that are disabled reduce a defensive layer against unmitigated cross-chain flows.

**Recommendation:** Enable TPRLs on **all lanes** for production CCT deployments. See [Rate Limit Management Overview](/ccip/concepts/rate-limit-management/overview).

<h3 id="misconfigured-token-pool-rate-limits">Misconfigured token pool rate limits (Critical)</h3>

When source **outbound** capacity exceeds destination **inbound** capacity, in-flight transactions can become stuck.

**Remediation:** Configure destination inbound capacity to **meet or exceed** source outbound capacity plus a buffer (typically 5–10%, or `SourceOutbound × 1.1`). See [Configuration Checklist](/ccip/concepts/rate-limit-management/configuration-checklist).

<h3 id="unbuffered-token-pool-rate-limits">Unbuffered token pool rate limits (Warning)</h3>

Inbound limits that ignore refill rate and lane latency can cause stuck in-flight transactions.

**Remediation:** Set `DestinationInboundCapacity ≥ SourceOutboundCapacity × 1.1`, accounting for source chain epoch size and CCIP round time. See [Configuration Checklist](/ccip/concepts/rate-limit-management/configuration-checklist).

<h3 id="token-bucket-refill-rate">Token bucket refill rate (Warning)</h3>

Refill rates that are too slow limit user availability; rates that are too fast fail to throttle abnormal flows.

**Guideline:** Target a full-bucket refill time between **1 and 24 hours**. See [Refill Rate Guidance](/ccip/concepts/rate-limit-management/common-scenarios#refill-rate-guidance).

<h3 id="paused-token-pool-rate-limits">Paused token pool rate limits (Warning)</h3>

Rate limits configured with minimal capacity and refill (`capacity: 1`, `rate: 1`) effectively pause a lane on many EVM TokenPool versions. This may be intentional during incidents or accidental from misconfiguration. On **1.4.0 – 1.6.0** pools, that exact config reverts at setup — see [Contract Version Behavior (EVM)](/ccip/concepts/rate-limit-management/contract-version-behavior).

**Remediation:** Distinguish intentional pauses ([Emergency Actions](/ccip/concepts/rate-limit-management/emergency-actions)) from misconfiguration using the [Configuration Checklist](/ccip/concepts/rate-limit-management/configuration-checklist).

<h3 id="in-flight-transactions-and-tprls">In-flight transactions and TPRLs (Warning)</h3>

Reducing inbound capacity while messages are in-flight can cause them to stall until capacity is available.

**Runbook:**

1. Set outbound capacities first.
2. Check [CCIP Explorer](/ccip/tools-resources/ccip-explorer) or the CCIP API for in-flight messages whose amounts exceed the new outbound limits.
3. Wait for those transactions to settle on destination chains.
4. Then reduce inbound capacities.

See [Reducing Inbound Capacity](/ccip/concepts/rate-limit-management/update-rate-limits#reducing-inbound-capacity).

<h3 id="suggested-tprls">Suggested TPRLs (Info)</h3>

Balance two goals: maximum transfer availability for end users and minimum risk exposure through throttling. Use worked examples as a **starting point**, not a one-size-fits-all default.

See [Volume-Based Strategies](/ccip/concepts/rate-limit-management/common-scenarios#volume-based-strategies) and [Common Scenarios](/ccip/concepts/rate-limit-management/common-scenarios).

## Administration and permissions

<h3 id="pending-token-administrator">Pending token administrator (Warning)</h3>

A proposed but unaccepted token administrator can be accepted by the proposed address at any time.

**Remediation (EVM):** Cancel a pending transfer with `transferAdminRole(localToken, address(0))` on the TokenAdminRegistry. See [Transfer Administrator Role](/ccip/concepts/cross-chain-token/evm/registration-administration#transfer-administrator-role).

**Remediation (SVM):** Use [`owner_override_pending_administrator`](/ccip/concepts/cross-chain-token/svm/registration-administration#overriding-pending-administrator) before the pending admin accepts.

<h3 id="pending-owner">Pending owner (Warning)</h3>

Contracts using Ownable2Step (Token, Token Pool, AdvancedPoolHooks, ERC20LockBox) may have a pending owner awaiting acceptance.

**Remediation:** Cancel with `transferOwnership(address(0))` if the transfer was initiated in error.

<h3 id="burn-mint-rights">Burn/mint rights (Warning)</h3>

Only the active token pool and owner-like roles (`owner()`, `default_admin`) should hold mint and burn permissions.

**Remediation:** Audit all minter/burner roles and revoke unnecessary grants. See [BurnMint Requirements](/ccip/concepts/cross-chain-token/evm/tokens#burnmint-requirements).

## Upgrades and execution

<h3 id="stale-remote-pool-addresses">Stale remote pool addresses (Warning)</h3>

During zero-downtime pool upgrades, multiple remote pool addresses may be configured temporarily. After migration, remove stale addresses via [`removeRemotePool`](/ccip/concepts/cross-chain-token/evm/upgradability#6-clean-up).

**Risk:** Old pools may contain faulty or unintended logic and continue accepting in-flight messages.

**Before removal:** Confirm no pending or failed transactions reference the old pool in [CCIP Explorer](/ccip/tools-resources/ccip-explorer).

<h3 id="invalid-lane-configuration">Invalid lane configuration (Warning)</h3>

Remote pool configuration pointing to a chain selector not supported by the CCIP router causes transactions to revert.

**Remediation:** Validate chain selectors against the [CCIP Directory](/ccip/directory) before calling `applyChainUpdates`. Remove unintended remote chains from the pool.

<h3 id="destgasoverhead-configuration">destGasOverhead configuration (Warning)</h3>

Inconsistent `destGasOverhead` values across lanes can cause token transfers to fail automatic execution or overcharge fees.

**Remediation:**

- Test that token transfers execute automatically on every lane.
- Ensure gas covers first-time receivers (new wallets or accounts that have never held the token).
- Review [network-specific guides](/ccip/tools-resources/network-specific) for chains with non-default gas behavior (for example, [Tempo](/ccip/tools-resources/network-specific/tempo-integration-guide)).

See [destGasOverhead and Transfer Fees](/ccip/concepts/cross-chain-token/evm/token-pools#destgasoverhead-and-transfer-fees).

<h3 id="network-specific-considerations">Network-specific considerations (Warning)</h3>

Some networks have unique configuration requirements, RPC behavior, or service limits.

**Remediation:** Review [Network Specific Guides](/ccip/tools-resources/network-specific) and [Service Limits](/ccip/service-limits) before deploying to any chain.

## Architecture and versioning

<h3 id="liquidity-architecture">Liquidity architecture (Info)</h3>

| Complexity | Pattern                                | Notes                                       |
| ---------- | -------------------------------------- | ------------------------------------------- |
| Low        | Burn and Mint on all chains            | Simplest accounting; recommended default    |
| Medium     | Lock and Mint                          | Requires supply monitoring on issuing chain |
| High       | Lock and Unlock across multiple chains | Fragmented liquidity; not recommended       |

See [Liquidity Architecture Complexity](/ccip/concepts/cross-chain-token/evm/tokens#liquidity-architecture-complexity).

<h3 id="use-latest-tokenpool-versions">Use latest TokenPool versions (Info)</h3>

Older TokenPool versions have known limitations in decimal handling, TPRL enforcement, and upgradeability. Pre-1.5 proxy pools increase management complexity.

**Remediation:** Upgrade to the latest available TokenPool release when feasible. See [Contract Version Behavior (EVM)](/ccip/concepts/rate-limit-management/contract-version-behavior), [Token Pool Version Considerations](/ccip/concepts/cross-chain-token/evm/token-pools#token-pool-version-considerations), and [Upgradability (EVM)](/ccip/concepts/cross-chain-token/evm/upgradability).
