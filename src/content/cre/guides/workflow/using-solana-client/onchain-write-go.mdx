This guide walks through writing data from a CRE workflow to a Solana program using the Go SDK. By the end, you will have a working workflow that Borsh-encodes a payload, builds a forwarder report, and submits it with `solana.Client.WriteReport()`.

The pasteable example writes to a Devnet receiver you configure with the placeholders below. [Step 3](#step-3-run-the-simulation) exercises compile, trigger, and `WriteReport` against the CLI **mock** forwarder. [Step 4](#step-4-deploy-for-the-keystone-path) deploys so the DON uses **Keystone**.

<Aside type="note" title="Your own program and bindings">
  When you have your own Anchor IDL, generate typed helpers with [`cre generate-bindings
  solana`](/cre/guides/workflow/using-solana-client/generating-bindings-go) and call the generated `WriteReportFrom...`
  methods instead of the low-level `WriteReport` flow below.
</Aside>

## Prerequisites

- A CRE Go project initialized with `cre init`. See [Part 1: Project Setup](/cre/getting-started/part-1-project-setup-go) if you are starting from scratch.
- A funded Solana Devnet keypair for `CRE_SOLANA_PRIVATE_KEY` (required even for dry-run). Generate one with `solana-keygen new`, fund it with `solana airdrop` (or a transfer), and put the 64-byte base58 keypair in your project `.env`. See [Step 3](#step-3-run-the-simulation).

## What you need: a receiver program

Your Solana program must implement an `on_report` instruction that accepts the payload delivered by the forwarder. The forwarder CPIs into your program after verifying the DON signatures.

The example receiver in this guide expects a small Borsh payload: an `i64` little-endian Unix timestamp followed by a fixed 16-byte label. Your own program will define a different payload shape.

<Aside type="note" title="Forwarder strips leading accounts">
  The forwarder passes only indices 2+ of your `accounts` slice to your program's CPI call. Indices 0 (`forwarderState`)
  and 1 (`forwarderAuthority`) are consumed by the forwarder itself.
</Aside>

## Step 1: Configure the workflow

1. Open (or create) `config.staging.json` and add the Solana Devnet dry-run parameters.

   Dry-run (`cre workflow simulate`) always invokes the CLI **mock forwarder**. The mock `forwarderProgramId` and
   `forwarderState` below are fixed Devnet values baked into the CLI. You supply `receiverProgramId`, a derived
   `forwarderAuthority`, `reportState`, and `label`. Local simulate does not use production Keystone; see
   [Step 3](#step-3-run-the-simulation).

   ```json
   {
     "schedule": "*/30 * * * * *",
     "chainSelector": "16423721717087811551",
     "receiverProgramId": "<RECEIVER_PROGRAM_ID>",
     "forwarderProgramId": "7kuEAA3mSC1Tz8gQjnvH7bKFda9xSPRRin9SZbH49cNK",
     "forwarderState": "5Tipz3yhTBdVsDbaBxZkrp7Gjf3brGq5SKkxReefPMP7",
     "forwarderAuthority": "<MOCK_FORWARDER_AUTHORITY>",
     "reportState": "<REPORT_STATE>",
     "label": "<LABEL>",
     "computeLimit": 290000
   }
   ```

   | Field                | Description                                                                                                            |
   | -------------------- | ---------------------------------------------------------------------------------------------------------------------- |
   | `chainSelector`      | Solana Devnet (`"16423721717087811551"`). Store as a string to avoid JSON precision loss.                              |
   | `receiverProgramId`  | Your receiver program on Devnet (you supply)                                                                           |
   | `forwarderProgramId` | CLI mock forwarder program (dry-run only; fixed)                                                                       |
   | `forwarderState`     | Mock forwarder state account (dry-run only; fixed)                                                                     |
   | `forwarderAuthority` | PDA from `["forwarder", forwarderState, receiverProgram]` under the **mock** forwarder program ID (you derive)         |
   | `reportState`        | Writable account required by your receiver (you supply)                                                                |
   | `label`              | Fixed 16-byte string the example payload encodes (padded with zeros). Receiver-specific data; not your CRE DON Family. |
   | `computeLimit`       | Solana compute units; must be greater than 0 for simulation                                                            |

1. Derive `forwarderAuthority` and paste it into the config. Seeds are `[]byte("forwarder")`, forwarder state bytes, and
   receiver program bytes under the **mock** forwarder program ID above. When you move to Keystone in Step 4, derive
   again under the Keystone program ID.

   ```go
   import solanago "github.com/gagliardetto/solana-go"

   forwarderProgram := solanago.MustPublicKeyFromBase58(forwarderProgramId)
   forwarderState := solanago.MustPublicKeyFromBase58(forwarderState)
   receiverProgram := solanago.MustPublicKeyFromBase58(receiverProgramId)
   forwarderAuthority, _, err := solanago.FindProgramAddress(
   	[][]byte{
   		[]byte("forwarder"),
   		forwarderState[:],
   		receiverProgram[:],
   	},
   	forwarderProgram,
   )
   ```

## Step 2: Write the workflow

1. Save the following as `my-workflow/workflow.go`.

   A Go project is split across two files: `main.go` is the WASM entry point (generated by `cre init` and unchanged), and your workflow logic lives in a separate file such as `workflow.go`.

   ```go
   // my-workflow/workflow.go
   package main

   import (
   	"encoding/binary"
   	"log/slog"

   	solanago "github.com/gagliardetto/solana-go"
   	sdk "github.com/smartcontractkit/chainlink-protos/cre/go/sdk"
   	solana "github.com/smartcontractkit/cre-sdk-go/capabilities/blockchain/solana"
   	"github.com/smartcontractkit/cre-sdk-go/capabilities/blockchain/solana/bindings"
   	"github.com/smartcontractkit/cre-sdk-go/capabilities/scheduler/cron"
   	"github.com/smartcontractkit/cre-sdk-go/cre"
   )

   const (
   	labelLen            = 16
   	defaultComputeLimit = 290_000
   	defaultLabel        = "example-label"
   )

   type Config struct {
   	Schedule           string `json:"schedule"`
   	ChainSelector      uint64 `json:"chainSelector,string"`
   	ReceiverProgramID  string `json:"receiverProgramId"`
   	ForwarderState     string `json:"forwarderState"`
   	ForwarderAuthority string `json:"forwarderAuthority"`
   	ReportState        string `json:"reportState"`
   	Label              string `json:"label"`
   	ComputeLimit       uint32 `json:"computeLimit"`
   }

   func InitWorkflow(config *Config, logger *slog.Logger, secretsProvider cre.SecretsProvider) (cre.Workflow[*Config], error) {
   	cronTrigger := cron.Trigger(&cron.Config{Schedule: config.Schedule})

   	return cre.Workflow[*Config]{
   		cre.Handler(cronTrigger, onCronTrigger),
   	}, nil
   }

   func configuredLabel(cfg *Config) string {
   	if cfg.Label != "" {
   		return cfg.Label
   	}
   	return defaultLabel
   }

   func padLabel(label string) [labelLen]byte {
   	var z [labelLen]byte
   	copy(z[:], []byte(label))
   	return z
   }

   func encodeReportPayload(scheduledUnix int64, label string) []byte {
   	out := make([]byte, 8+labelLen)
   	binary.LittleEndian.PutUint64(out[0:8], uint64(scheduledUnix))
   	z := padLabel(label)
   	copy(out[8:], z[:])
   	return out
   }

   func scheduledUnixFromPayload(trigger *cron.Payload) int64 {
   	if ts := trigger.GetScheduledExecutionTime(); ts != nil {
   		return ts.AsTime().Unix()
   	}
   	return 0
   }

   func onCronTrigger(config *Config, runtime cre.Runtime, trigger *cron.Payload) (string, error) {
   	forwarderStatePk := solanago.MustPublicKeyFromBase58(config.ForwarderState)
   	forwarderAuthPk := solanago.MustPublicKeyFromBase58(config.ForwarderAuthority)
   	reportStatePk := solanago.MustPublicKeyFromBase58(config.ReportState)
   	receiverPk := solanago.MustPublicKeyFromBase58(config.ReceiverProgramID)

   	// Order matters for the account hash: 0 = forwarderState, 1 = forwarderAuthority, 2+ = receiver accounts.
   	accounts := []*solana.AccountMeta{
   		{PublicKey: forwarderStatePk[:], IsWritable: false},
   		{PublicKey: forwarderAuthPk[:], IsWritable: false},
   		{PublicKey: reportStatePk[:], IsWritable: true},
   	}

   	scheduledUnix := scheduledUnixFromPayload(trigger)
   	label := configuredLabel(config)
   	inner := encodeReportPayload(scheduledUnix, label)

   	runtime.Logger().Info("Submitting report", "label", label, "scheduledUnix", scheduledUnix)

   	accountHash := bindings.CalculateAccountsHash(accounts)
   	fwdReport := bindings.ForwarderReport{
   		AccountHash: accountHash,
   		Payload:     inner,
   	}
   	encoded, err := fwdReport.Marshal()
   	if err != nil {
   		return "", err
   	}

   	report, err := runtime.GenerateReport(&sdk.ReportRequest{
   		EncodedPayload: encoded,
   		EncoderName:    "solana",
   		HashingAlgo:    "keccak256",
   		SigningAlgo:    "ecdsa",
   	}).Await()
   	if err != nil {
   		return "", err
   	}

   	computeLimit := config.ComputeLimit
   	if computeLimit == 0 {
   		computeLimit = defaultComputeLimit
   	}
   	client := &solana.Client{ChainSelector: config.ChainSelector}
   	result, err := client.WriteReport(runtime, &solana.WriteCreReportRequest{
   		Receiver:          receiverPk[:],
   		RemainingAccounts: accounts,
   		ComputeConfig:     &solana.ComputeConfig{ComputeLimit: computeLimit},
   		Report:            report,
   	}).Await()
   	if err != nil {
   		return "", err
   	}

   	runtime.Logger().Info("Write completed",
   		"txStatus", result.GetTxStatus().String(),
   		"errorMessage", result.GetErrorMessage(),
   		"txSignature", result.GetTxSignature(),
   	)
   	if msg := result.GetErrorMessage(); msg != "" {
   		return msg, nil
   	}
   	return "Success", nil
   }
   ```

## Step 3: Run the simulation

Local simulate always uses the CLI mock forwarder from Step 1 (not Keystone). Use this step to confirm compile, trigger,
and `WriteReport` wiring.

1. Add a Solana RPC endpoint to your `project.yaml` under your simulation target.

   ```yaml
   staging-settings:
     rpcs:
       - chain-name: solana-devnet
         url: https://api.devnet.solana.com
   ```

1. Set `CRE_SOLANA_PRIVATE_KEY` in your project `.env`.

   Solana simulation requires a 64-byte base58 keypair even for dry-run. The transmitter account must exist and be
   funded on Solana Devnet. Generate a key with `solana-keygen new`, fund it with `solana airdrop` (or a transfer), and
   put the keypair in `.env`:

   ```bash
   CRE_SOLANA_PRIVATE_KEY=<your-64-byte-base58-keypair>
   ```

1. Run the dry-run simulation.

   ```bash
   cre workflow simulate my-workflow --target staging-settings
   ```

   Confirm the workflow compiles, the cron trigger runs, and your USER LOGs appear:

   ```
   ✓ Workflow compiled
   [SIMULATION] Simulator Initialized
   [SIMULATION] Running trigger trigger=cron-trigger@1.0.0
   [USER LOG] msg="Submitting report" label=<LABEL> scheduledUnix=...
   [USER LOG] msg="Write completed" txStatus=... txSignature=...

   ✓ Workflow Simulation Result:
   ...
   ```

   For a Keystone write on Devnet, continue to [Step 4](#step-4-deploy-for-the-keystone-path).

## Step 4: Deploy for the Keystone path

A **deployed** workflow is what runs against production Keystone on Solana Devnet. Prefer the
[private registry](/cre/guides/operations/deploying-to-private-registry-go) for this example (CRE login session; no
Ethereum gas for registry ops). You need [Deploy Access](/cre/account/deploy-access).

1. Create `config.production.json` with Devnet Keystone forwarder addresses (same receiver and report state as staging).

   ```json
   {
     "schedule": "30 */5 * * * *",
     "chainSelector": "16423721717087811551",
     "receiverProgramId": "<RECEIVER_PROGRAM_ID>",
     "forwarderProgramId": "<KEYSTONE_FORWARDER_PROGRAM_ID>",
     "forwarderState": "<KEYSTONE_FORWARDER_STATE>",
     "forwarderAuthority": "<KEYSTONE_FORWARDER_AUTHORITY>",
     "reportState": "<REPORT_STATE>",
     "label": "<LABEL>",
     "computeLimit": 290000
   }
   ```

   `forwarderAuthority` is the PDA from `["forwarder", forwarderState, receiverProgram]` under the **Keystone**
   forwarder program ID (re-derive with the Step 1 snippet). Get Keystone addresses from your Chainlink representative
   or from [`cre workflow supported-chains`](/cre/reference/cli/workflow#cre-workflow-supported-chains) after Solana is
   enabled for your organization.

1. Point a deploy target at that config and the private registry in `my-workflow/workflow.yaml`.

   ```yaml
   production-settings:
     user-workflow:
       workflow-name: "my-solana-write"
       deployment-registry: "private"
     workflow-artifacts:
       workflow-path: "./main.go"
       config-path: "./config.production.json"
       secrets-path: ""
   ```

1. Add the Solana Devnet RPC under `production-settings` in `project.yaml` (same URL as staging is fine).

1. Deploy from your project root.

   ```bash
   cre workflow deploy my-workflow --target production-settings
   ```

   Expected shape:

   ```
   Deploying Workflow: my-solana-write
     Registry:      private
     ...
   ✓ Workflow registered in private registry
   ...
        Status:           Active
   ```

1. After the cron fires, open the workflow in the [CRE platform](https://app.chain.link/cre/workflows) (or run
   `cre execution list my-solana-write`) and confirm node logs reach the write path:

   ```
   Submitting report label=<LABEL> scheduledUnix=...
   ```

   If WriteReport returns a Solana `txSignature`, confirm the transaction on
   [Solana Explorer](https://explorer.solana.com/?cluster=devnet) (forwarder as the top-level program, your receiver as
   a CPI).

See [Deploying to the Private Registry](/cre/guides/operations/deploying-to-private-registry-go) for registry details, activate/pause, and CI flags.

## What happens onchain

When the deployed workflow runs on the DON:

1. The DON nodes each execute your workflow and reach consensus on the encoded payload.
1. A DON node calls the Keystone Forwarder program with the signed report and account list.
1. The forwarder verifies all DON signatures.
1. The forwarder CPIs into your program's `on_report` instruction with the decoded payload and receiver accounts.

The transaction on Solana Explorer will show the forwarder program as the initial callee, with your program appearing as a CPI target.
